### **SONNET PROMPT #1.2: Path Traversal Fix**
```
Critical security fix in packages/local-bridge/src/ws/server.ts handleFileEdit().

Current vulnerable code (line ~493):
const absPath = resolve(join(this.options.repoRoot, relPath));
if (!absPath.startsWith(this.options.repoRoot)) { ... }

Fix:
1. Normalize relPath first: remove all .. sequences and null bytes
2. Use path.join (not resolve) to prevent absolute path injection
3. Verify final path starts with resolved repoRoot AFTER normalization

Implementation:
- Create src/utils/path-sanitizer.ts with sanitizeRepoPath(relPath: string, repoRoot: string): string
- Must handle: ../../../etc/passwd, file\x00.txt, file..txt, /absolute/path
- Throw SanitizationError with specific message for logging

Update handleFileEdit to use this utility. Add test cases for each attack vector.
```

### **OPUS PROMPT #2: Public Digest Architecture**
```
Design the auto-publication system for cocapn. Users have private repos (brain) and 
public repos (devlog). We need to automatically generate sanitized content from private 
work for public consumption.

Design:

1. New module type: "publisher" in packages/modules/publisher/
   - Hooks into git commit events
   - Reads from Brain (tasks, wiki)
   - Generates sanitized summaries
   - Writes to public repo

2. Sanitization strategy:
   - Remove specific code snippets (keep descriptions)
   - Remove secret-containing files
   - Strip internal URLs/IPs
   - Keep: project names, tech stack mentions, problem-solution narratives

3. Content pipeline:
   - Commit to private repo → trigger analysis
   - Diff extraction: what changed?
   - Summarization: local LLM call (claude-code via MCP) to generate public-safe text
   - Commit to public repo: markdown file in updates/YYYY-MM-DD.md

4. Streak tracking:
   - Track consecutive days with commits
   - Store in public repo stats.json
   - Display in UI

Provide:
- Data flow diagram (ASCII art fine)
- Interface for Sanitizer class
- Storage format for public updates
- Hook points in existing GitSync class
```

### **OPUS PROMPT #3: Discovery & Federation**
```
Design the social layer for cocapn fleet. Currently fleet is private (your devices).
We want opt-in public discovery.

Requirements:
1. Public profiles stored in each user's public repo: cocapn/profile.json
   - username, displayName, bio, currentFocus (from facts.json key "current-project")
   - avatar URL (Gravatar or GitHub)
   - domains: string[] (other cocapn instances they run)

2. AdmiralDO extension (cloud-agents/src/admiral.ts):
   - New endpoint: POST /registry/register
     - Accepts profile.json + proof (signature from fleet key)
     - Stores in Durable Object storage
   - New endpoint: GET /registry/discover?query=
     - Full-text search across registered profiles
     - Returns paginated list

3. Privacy controls:
   - Profile only published if public_repo/cocapn/config.yml has discovery: true
   - Users can block specific domains
   - Facts prefixed with "private." never exported

4. Cross-domain messaging:
   - User A on makerlog.ai can ask User B on devlog.ai
   - Routed through AdmiralDO (message queue)
   - Or direct A2A if both online (WebRTC or HTTP)

Design the trust model:
- How do we verify identity? (GitHub OAuth + fleet key signature)
- How do we prevent spam? (rate limiting, mutual follows only)
- How do we handle offline messages? (AdmiralDO queue with TTL)

Provide schema definitions and API specs.
```

## Phase 4: Cloud-Native & Scheduling

---

### **OPUS PROMPT #4: Background Agent System**
```
Design the 24/7 background agent capability using Cloudflare Workers.

Current state: CloudAdapter sends tasks to workers, but only when local bridge triggers.

Desired state:
1. Scheduled tasks: User creates task in cocapn/tasks/scheduled/check-logs.md
   - Contains cron expression in frontmatter: `cron: "0 */6 * * *"`
   - Contains instructions for agent

2. AdmiralDO manages schedule:
   - Parses cron expressions
   - Triggers Cloud Worker at scheduled time
   - Worker pulls task context from GitHub (using user's PAT)
   - Executes, commits results back to private repo
   - Notifies local bridge via WebSocket (if online)

3. Trigger types:
   - Cron (time-based)
   - Webhook (GitHub issue labeled "urgent")
   - Event (new commit to specific file)

4. Security:
   - Worker only gets encrypted secrets (age-encrypted, decrypted by worker)
   - Worker identity verified via Cloudflare Access + fleet key
   - Audit log of all cloud actions in cocapn/audit-cloud.log

Design the cron parser, the task manifest format, and the Worker<->Admiral handshake.
```

### **SONNET PROMPT #4.2: Cloud Worker Agent**
```
Update packages/cloud-agents/src/worker.ts (or create if doesn't exist).

The worker handles HTTP requests from AdmiralDO:

Endpoints:
1. POST /task/scheduled
   - Body: { taskId, instructions, repoUrl, encryptedSecrets }
   - Decrypts secrets using worker's age key (from env SECRET_KEY)
   - Clones repo (shallow) using provided GitHub token
   - Spawns agent process (claude-code or other via MCP)
   - Runs instructions
   - Captures output
   
2. POST /task/webhook
   - GitHub webhook handler
   - Validates signature (GITHUB_WEBHOOK_SECRET)
   - Routes to appropriate task based on config

3. GET /health
   - Returns current load, queued tasks

Execution:
- Use Cloudflare Worker's waitUntil for background processing
- Stream logs to AdmiralDO via POST /log
- Commit results back to repo using GitHub API (create commit)
- Handle timeouts (max 30s CPU for Workers, use Durable Objects for longer)

Error handling:
- Retry 3x with backoff
- On failure, create issue in repo or notify via email (SendGrid integration?)

Configuration via cocapn-cloud.yml in private repo.
```
